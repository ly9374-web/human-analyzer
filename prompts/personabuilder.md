# 人物侧写生成提示词

你需要根据 `relationship.md` 和 `memory.md`，并结合用户提供的目标人物核心身份与背景，为目标人物生成 `persona.md`。这个文件不是给用户的回复，而是一份稳定、可调用的人物侧写，供未来的 `{slug}/SKILL.md` 在回答关于此人的问题前读取。

`persona.md` 的输入仅为：

- `relationship.md`
- `memory.md`
- 用户提供的目标人物核心身份与背景
- 目标人物姓名或别名
- 用户输入记录（仅当本次材料最终只生成 `chunk_001.md` 一个 chunk 时提供）

默认不要直接使用原始聊天记录、截图、解析后的 JSON、chunk 文件、中间稿或其他未被写入最终 `relationship.md` / `memory.md` 的材料。除非用户在本轮明确补充核心身份与背景，否则不要把外部信息写入 `persona.md`。

单 chunk 例外：如果调用方明确说明本次材料最终只生成 `chunk_001.md` 一个 chunk，并提供用户输入记录，则可以把这份用户输入记录作为补充上下文。这里的用户输入记录应是进入 `chunk_text.py` 前的可读材料：聊天导出使用解析后的可读记录或完整 `chunk_001.md` 内容；普通文本、笔记、截图提取文本使用原始可读文本或完整 `chunk_001.md` 内容。如果没有明确提供单 chunk 用户输入记录，不要假设或补写其中内容。

默认使用与输入材料相同的语言。如果材料主要是中文，就用中文撰写。

## 生成目标

你需要根据 `relationship.md` 和 `memory.md` 得出（但在输出中不要提到证据和数据来源）：

- 通用性格侧写
- 常见决策风格
- 沟通风格
- 喜好
- 厌恶
- 冲突时的风格：冲突时会如何对冲突进行反应

用户会提供目标人物的核心身份与背景；该部分只整理用户明确提供的内容，不从关系策略或记忆中强行推断身份事实。

## 生成边界

每个实质性判断都必须基于 `relationship.md`、`memory.md`、用户明确提供的核心身份与背景，或单 chunk 例外中提供的用户输入记录，但最终输出里不要写证据字段、来源字段或来源说明。

区分用户明确提供的背景、关系侧写中呈现的互动模式、记忆档案中呈现的事件与偏好、单 chunk 用户输入记录中补充或校正的信息，以及综合材料得到的推断。单 chunk 用户输入记录可以补充和校正 `relationship.md`、`memory.md` 中没有完整保留的细节；如果它与 `relationship.md`、`memory.md` 冲突，要明确描述材料冲突或降低置信度，不要悄悄覆盖。不要把薄弱材料写成事实。不要诊断临床状况。不要使用 MBTI、依恋风格、大五人格等框架，除非用户明确提供；即便用户提供，也只能作为“用户提供标签”而不是分析结论。

## 分级规则

“喜好”和“厌恶”如果能从材料中得出强弱层级，则给出分级；如果不能得出分级，必须明确写“无法分级：现有材料只能证明存在该倾向，不能比较强弱”。

可使用的分级方式：

- 高/中/低：当不同喜好或厌恶有明显数量差异、反复程度差异或影响强度差异时使用。
- 核心/明显/轻微：当材料更适合描述重要程度，而不是频率时使用。
- 无法分级：当材料不足以比较强弱时使用。

每个分级都必须基于 `relationship.md`、`memory.md` 或明确提供的单 chunk 用户输入记录中的内容。不要为了完整好看而强行分级。

## 输出格式

按以下结构把最终文件写成 `persona.md`：

````md
# 人物侧写：[目标姓名]

## 侧写状态
- 目标：[姓名或别名]

## 1. 核心身份与背景

只整理用户明确提供的目标人物核心身份与背景。不要从 `relationship.md` 或 `memory.md` 反推身份事实。

```yaml
身份与背景: ""
置信度: "高/中/低"
```

## 2. 通用性格侧写

从 `relationship.md` 和 `memory.md` 中提炼跨关系、跨事件重复出现的稳定特征。优先写具体行为倾向，不要堆抽象标签。

```yaml
结论: ""
表现: ""
置信度: "高/中/低"
```

## 3. 常见决策风格

描述目标人物通常如何做选择、推进事情、权衡风险、改变主意、寻求确认、拖延或行动。没有足够材料时留空，不写泛泛判断。

```yaml
结论: ""
典型场景: ""
置信度: "高/中/低"
```

## 4. 沟通风格

描述目标人物常见的回复方式、语气、直接程度、细节量、节奏、情绪表达方式、请求方式和反馈方式。

```yaml
结论: ""
表现: ""
适用边界: "说明该沟通风格适用于哪些关系或场景；如果是通用风格则写通用。"
置信度: "高/中/低"
```

## 5. 喜好

列出目标人物喜欢、偏好、容易正向响应或更愿意投入的事物、表达方式、互动方式、安排方式。能分级则分级，不能分级则说明不能分级。

```yaml
分级方式: "高/中/低、核心/明显/轻微，或无法分级"
分级说明: ""
条目:
  - 喜好: ""
    分级: ""
    置信度: "高/中/低"
```

## 6. 厌恶

列出目标人物反感、排斥、容易退缩、压力上升、兴趣下降或产生防御的事物、表达方式、互动方式、安排方式。能分级则分级，不能分级则说明不能分级。

```yaml
分级方式: "高/中/低、核心/明显/轻微，或无法分级"
分级说明: ""
条目:
  - 厌恶: ""
    分级: ""
    置信度: "高/中/低"
```

## 7. 冲突时的风格

描述目标人物冲突时会如何对冲突进行反应，包括可能的防御、沉默、解释、退缩、反击、降温、修复、求确认或转移话题等。必须基于 `relationship.md` 的冲突预测和 `memory.md` 的冲突记忆。

```yaml
结论: ""
冲突触发: ""
典型反应: ""
化解方向: "基于 relationship.md 和 memory.md 写如何更可能让对方消气、恢复沟通或降低防御。"
置信度: "高/中/低"
```
````

实际生成时不要保留空字符串模板。没有足够材料的栏目保留标题，但写“现有可用材料不足以得出可靠结论”。

## 质量标准

- 只从 `relationship.md`、`memory.md`、用户提供的核心身份与背景，以及明确提供的单 chunk 用户输入记录生成。
- 优先写具体行为，而不是抽象标签。
- 如果材料冲突，描述分歧，不要强行合并成一个结论。
- 喜好和厌恶可以分级才分级，不能分级必须明确说明。
- 冲突风格必须写“会如何反应”，并在材料支持时写“如何化解冲突”。
- 让文件对未来回答关于此人的问题有用，而不是作为单篇文章显得华丽。
