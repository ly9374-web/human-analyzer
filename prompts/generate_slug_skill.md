# 最终人物技能生成提示词

在已经为目标人物生成 `persona.md`、`relationship.md` 和 `memory.md` 之后，使用本提示词。这里的 `persona.md` 应已经拼接了 `thinslice.md` 的内容；最终技能入口不需要额外读取 `thinslice.md`。

你的任务是为此人生成最终的 Codex 技能入口文件。推荐输出路径是：

```text
partners/{slug}/SKILL.md
```

如果用户特别想要单个命名文件，也可以提到 `{slug}skill.md`，但要说明 Codex 技能发现机制期望的是包含标准 `SKILL.md` 文件的文件夹。文件头元数据中的 `name` 必须正好是 `{slug}`，这样技能才能通过以下方式调用：

```text
/{slug}
```

## 变量

调用流程会提供：

- `{slug}`：机器可读的技能名称。尽可能使用小写字母、数字和连字符。
- `{display_name}`：目标人物的人类可读姓名或别名。
- `{persona_md}`：已生成的人物侧写内容，包含已经拼接进去的薄片心理侧写。
- `{relationship_md}`：已生成的互动策略内容。
- `{memory_md}`：已生成的长期记忆档案内容。
- `{user_extra_info}`：用户补充信息，如有。

## 核心原则

生成一个紧凑的控制器技能。不要把完整的 `persona.md`、`relationship.md` 和 `memory.md` 粘贴进最终 `SKILL.md`。

最终 `SKILL.md` 应该：

1. 给出简短核心侧写摘要，可从 `persona.md` 中已拼接的薄片心理侧写提取稳定、高价值判断。
2. 告诉 Codex 何时读取每个本地支持文件。
3. 指示 Codex 以本地档案为回答基础。
4. 说明如何回应用户。
5. 解释 Codex 技能用法与普通 API/网页聊天用法的区别。
6. 当支持文件没有足够材料时，避免过度自信的判断。

## 必须包含的文件读取规则

生成的 `SKILL.md` 必须包含这些规则：

- 在回答关于 `{display_name}` 的问题前，总是先读取 `persona.md`。
- 当用户询问沟通、互动策略、关系动态、反馈、请求、节奏、修复、拒绝、协商或边界设定时，读取 `relationship.md`。
- 当用户询问过去事件、长期偏好、重要经历、重复模式、具体细节，或某件事是否符合此人时，读取 `memory.md`。
- 对复杂问题，按以下顺序综合：

```text
persona -> relationship -> memory
```

- 如果相关文件不支持某个判断，就说明不确定之处，不要编造。
- 当本地侧写文件包含具体材料时，不要只凭通用心理学回答。
- 可以在材料不充分时使用心理学理论推断，但要提到在推断后加入明文指出这一段为心理学理论推断，置信度低

## 必须生成的最终输出

只生成最终 `SKILL.md` 的内容。前后不要包含解释性评论。

最终文件必须使用以下文件头元数据，并填入变量：

```yaml
---
name: {slug}
description: "{display_name} 的个人侧写与互动参考技能。用于根据 persona、relationship、memory 三个本地档案回答关于此人的表达风格、互动方式、说话风格、偏好、边界和长期模式的问题。"
user-invocable: true
---
```

然后按以下结构撰写正文：

````md
# {display_name}

这个技能为 `{display_name}` 提供一份基于本地档案的参考。它帮助回答关于此人的表达风格、说话风格、互动模式、偏好、边界、重复行为，以及有效沟通方式的问题。

## 可用文件

- `persona.md` — 稳定个人侧写：表达风格、人格结构、偏好、边界、反应模式、稳定特质和已拼接的薄片心理侧写。
- `relationship.md` — 互动策略：沟通建议、说话风格、反馈、节奏、修复、请求、拒绝、协商和边界设定。
- `memory.md` — 长期档案：过去事件、重要经历、重复模式、偏好、说话风格、常用表达和具体事实核查。

## 核心侧写摘要

根据 `{persona_md}`、`{relationship_md}`、`{memory_md}` 和 `{user_extra_info}`，写 4-8 条最稳定、最有用、最有材料依据的要点。

每条要短而实用。这里不需要逐条解释依据。

## 调用流程

当用户调用 `/{slug}` 或询问关于 `{display_name}` 的问题时：

1. 先读取 `persona.md`。
2. 判断用户问题是否还需要 `relationship.md`、`memory.md`，或两者都需要。
3. 对沟通、策略、反馈、请求、节奏、修复、拒绝、协商或边界问题，读取 `relationship.md`。
4. 对过去事件、长期偏好、重要经历、重复模式、具体细节或匹配度判断，读取 `memory.md`。
5. 复杂问题按以下顺序综合：`persona.md` -> `relationship.md` -> `memory.md`。
6. 用最具体的可用材料回答。如果文件不支持某个判断，就说明这一点。

## 回答风格

- 具体且直接有用。
- 当用户询问该说什么或该做什么时，优先给出实用措辞、话术或下一步。
- 让建议贴合 `{display_name}` 已记录的风格，而不是给通用人格建议。
- 区分观察到的模式、用户提供的上下文和推断。
- 置信度低时使用谨慎措辞。
- 如果用户要求写消息草稿，写成更可能适合 `{display_name}` 和用户真实上下文的风格。

## 材料纪律

- 不要编造原话、记忆、动机、偏好或诊断。
- 不要把推断当事实。
- 不要把此技能用于操纵、监视、欺骗或侵犯隐私。
- 如果用户请求的内容会施压、欺骗、测试或操纵 `{display_name}`，应转向诚实、尊重同意的沟通方式。
- 如果本地文件互相冲突，指出冲突，并基于最强材料回答。

## 输出模式

多数问题按以下形状回答：
1. 简短判断或解释。
2. 这对 `{display_name}` 意味着什么。
3. 推荐行动或措辞。（给出置信度）
4. 应避免什么。

除非用户要求完整分析，否则保持简洁。
````
